
# 【要件定義書】英会話スクリプト自動音声生成・学習アプリ

## 1. システム概要

本システムは、2人の対話（英会話スクリプト）のJSONデータを **ローカルのCLIスクリプト** で処理すると、AI音声合成（`kokoro-js`）によって**超高音質なネイティブ音声を自動生成**し、生成した音声ファイルとレッスンデータを**プロジェクト内（リポジトリ）に保存**する。
保存されたデータを元に Next.js アプリをビルドし、スクリプトのテキストと連動してリスニング・シャドーイング学習ができる自分専用のWebアプリである。

データベースや外部ストレージは持たず、レッスンデータと音声はすべてプロジェクト内のファイルとして管理する。レンダリング方式は静的生成（SSG）を基本としつつ、必要に応じてサーバーサイド処理（SSR / API Routes 等）も利用してよい（静的エクスポート専用に縛らない）。

---

## 2. テックスタック

* **フレームワーク**: Next.js (App Router)。レンダリング戦略は以下のハイブリッドとする。
* **SSG**: レッスン一覧・個別ページの骨格（テキスト・翻訳・音声パス等の静的コンテンツ）はビルド時に事前生成し、初期表示を高速にする。
* **CSR（Client Component）**: 音声プレイヤー、表示モード切替、行ハイライト、行タップ再生などのインタラクション部分はすべてクライアント側で動作させ、SPA のように軽快に操作できるようにする。
* 必要に応じて SSR / API Routes 等のサーバーサイド機能も利用可能とする（静的エクスポートには縛らない）。
* **インフラ（ホスティング）**: Next.js を実行できるホスティング（Vercel など）。静的ホスティング（Cloudflare Pages / GitHub Pages 等）への配置も可能だが必須要件ではない。
* **データ保存**: プロジェクト内のファイル（レッスンJSON ＋ 音声ファイル）。DB・外部ストレージは**使用しない**。
* **音声合成 (TTS)**: `kokoro-js`（Node.js 上で駆動）をローカルのCLIスクリプトから実行。完全無料・オフライン生成。
* **UIライブラリ**: Tailwind CSS + shadcn/ui

## 3. データ構造定義

### 3.1. 登録用JSON構造（インプット形式）

ローカルの入力用ディレクトリ（例: `content/input/`）に、レッスンごとに以下の構造のJSONファイルを配置する。

```json
{
  "title": "At the Cafe",
  "description": "Ordering a coffee and a pastry.",
  "category": "Daily Conversation",
  "turns": [
    {
      "turn_id": 1,
      "speaker": "Alice",
      "voice_setting": "af_bella",
      "text": "Hello! I'd like a double espresso, please.",
      "translation": "こんにちは！ダブルのエスプレッソをください。"
    },
    {
      "turn_id": 2,
      "speaker": "Bob",
      "voice_setting": "am_adam",
      "text": "Sure thing! Would you like anything else with that?",
      "translation": "かしこまりました！ご一緒にいかがですか？"
    }
  ]
}
```

> **補足**: `voice_setting` には `kokoro-js` が対応しているボイス名（例: `af_bella`＝アメリカ女性、`am_adam`＝アメリカ男性など）を指定し、2人の演じ分けを実現する。

### 3.2. プロジェクト内のデータ保存構造（生成物）

CLIスクリプト実行後、以下のようにプロジェクト内へ保存する（すべて Git にコミットする）。

```
content/
  input/                      # 入力用JSON（手動で配置）
    at-the-cafe.json
  lessons/                    # 生成済みレッスンメタデータ
    at-the-cafe.json          # 入力JSON ＋ 各turnの音声ファイルパスを付与したもの
public/
  audio/
    at-the-cafe/
      1.mp3                   # turn_id ごとの音声ファイル
      2.mp3
```

* **レッスンメタデータ（`content/lessons/<slug>.json`）**
* `id` / `slug`: レッスン識別子（ファイル名から導出）
* `title`: レッスンのタイトル
* `description`: 説明
* `category`: カテゴリ
* `turns[]`: 各発言（`turn_id`, `speaker`, `text`, `translation`, `audio_path`, `audio_hash`）
* `audio_path`: プロジェクト内の音声ファイルへの相対パス（例: `/audio/at-the-cafe/1.mp3`）
* `audio_hash`: その行の `text` ＋ `voice_setting` から算出したハッシュ値。差分再生成の判定に用いる（後述 4.1）。

> DB の `lessons` / `dialogues` テーブルは廃止し、上記のファイル構造に置き換える。

## 4. 機能要件

### 4.1. 管理機能（レッスン生成 — ローカルCLI）

* **入力JSONの配置**: `content/input/` に 3.1 の構造のJSONファイルを置く。
* **音声自動生成スクリプト**: `npm run generate`（例）でCLIスクリプトを実行する。
* `content/input/` 内のJSONを読み込む。
* **スラッグ導出ルール**: スラッグはファイル名から拡張子（`.json`）を除いた文字列をそのまま使用する。ファイル名にはスペースや記号を含めないことを前提とし、CLIによる自動変換は行わない（例: `at-the-cafe.json` → slug `at-the-cafe`）。
* **CLI進捗表示**: 処理中は `[1/5] turn 1 — generating...` のようにターン単位の進捗をコンソールに出力し、スキップ・生成・エラーをそれぞれ明示する。
* **入力バリデーション**: 音声生成の前に各レッスンを検証し、問題があれば**どのファイル・どの行か**を示してエラー終了する（不正なまま生成・出力しない）。少なくとも次を検証する。
* **話者は2人まで**: 1レッスンに登場する話者（`speaker`）の種類が**3種類以上**ある場合はバリデーションエラーとする（2人想定のため）。
* **話者名の表記統一**: 大文字小文字や前後の空白のみが異なる話者名（例: `"Alice"` と `"alice"`、`"Bob "` と `"Bob"`）が混在する場合は、表記ゆれによる別人誤判定を防ぐため**バリデーションエラー**とし、名称の統一を促す（CLIによる暗黙の正規化・補正は行わない）。
* 各 `turn` の `text` と `voice_setting` を元に、`kokoro-js`（Node.js）で1行ずつ音声ファイルを生成。
* 音声ファイルを **MP3** に変換し、`public/audio/<slug>/<turn_id>.mp3` として保存する。`kokoro-js` の出力（PCM/WAV）を MP3 にエンコードして容量を抑える。エンコードには **`fluent-ffmpeg` ＋ `ffmpeg-static`** を使用し、システムへの ffmpeg インストールを不要とする。
* **ターン単位の逐次処理**: 音声生成は1ターンずつ順番に実行する（並列化しない）。
* 入力JSONに各音声パスとハッシュ（`audio_hash`）を付与したメタデータを `content/lessons/<slug>.json` に出力。
* **ハッシュによる差分再生成**: 同じ入力JSONを更新しながら `generate` を繰り返し実行する運用を前提とする。各行の `text` ＋ `voice_setting` からハッシュを算出し、前回出力した `content/lessons/<slug>.json` の `audio_hash` と比較して再生成要否を判定する。
* ハッシュが一致する行（内容に変更なし）＝音声を再生成せずスキップ（高速）。
* ハッシュが異なる行（英文や声を修正）／新規の行＝その行だけ音声を再生成し、`audio_path` と `audio_hash` を更新する。
* メタデータが存在しない、または音声ファイルが欠落している行も生成対象とする。
* `--force` 等のオプションで全行を強制再生成できるようにする（任意）。
* 入力から削除された行に対応する不要な音声ファイル・メタデータは削除（またはクリーンアップ）する。
* **レッスンごとの削除**: `content/input/` から JSON ファイルごと削除した状態で `generate` を実行すると、対応する `content/lessons/<slug>.json` および `public/audio/<slug>/` ディレクトリ（配下の音声ファイルを含む）を自動的に削除する。
* **生成物のコミット**: 生成された音声ファイルとレッスンメタデータを Git にコミットし、リポジトリで履歴管理する。
* **認証は不要**: 生成はローカルでのみ行われ、本番には管理画面・APIを一切デプロイしないため、パスワード認証などは不要。

### 4.2. 学習機能（メイン画面 — 静的ページ）

* **レッスン一覧表示**: `content/lessons/` のメタデータをビルド時に読み込み、登録されたレッスン（タイトルやカテゴリ）をカード型で一覧表示。
* **並び順**: レッスンは**ファイル名（slug）の昇順**で並べる。これにより、入力JSONのファイル名（例: `01-...`, `02-...`）で表示順を意図的にコントロールできる。
* **カテゴリでのフィルター**: レッスンの `category` で絞り込みできる。カテゴリの選択UI（チップ／タブ／セレクト等）を用意し、選択したカテゴリのレッスンのみを表示する。「すべて」を選ぶと全件表示に戻せる。
* カテゴリ一覧はメタデータから自動的に収集する（重複は除く）。
* フィルターは CSR でクライアント側のみで動作させ、ページ遷移なしで即座に絞り込み結果へ切り替える。選択中のカテゴリは `localStorage` 等で保持し、再訪時も維持する。
* 絞り込み後も並び順はファイル名（slug）昇順を維持する。
* **会話スクリプト表示**:
* 選択したレッスンの会話を、チャットUIのように視覚的にわかりやすく表示する。会話は**2人の話者**を想定する。
* **話者の左右配置**: 左右は**話者（`speaker`）ごとに固定**する。レッスン内で**最初に登場した話者を左**、**次に登場したもう一方の話者を右**に割り当て、以降は各発言の `speaker` に応じて常に同じ側へ表示する（例: A さんは全行とも左、B さんは全行とも右）。
* 配置は発言の交互性ではなく**話者そのもの**で決まる。同じ話者が**2回以上連続**で話す場合も、その話者の側（例: A さんなら左）に連続して配置する（左右を交互に振らない）。
* **話者が1人のみの場合**: レッスンに1人しか登場しない（モノローグ）場合は、全行を**左**に配置する。
* **表示モード切り替え**: トグルボタンで以下の2つの表示モードを切り替えられる。
* **英文＋日本語訳モード**: 各行に英語テキストと日本語訳の両方を表示する。
* **英文のみモード**: 日本語訳を非表示にし、英語テキストのみを表示する（英語に集中するため）。
* 初期表示は「英文のみモード」とし、英語に集中できる環境をデフォルトとする。切り替え状態は学習中に保持する（例: `localStorage` に保存）。
* **再生中の切り替え**: 音声の再生を止めることなく、再生中でもいつでもモードを切り替えられる。切り替えは表示のみに作用し、再生位置・再生状態（再生中／一時停止）・ハイライト中の行には影響しない。
* **音声再生プレイヤー**: 操作はシンプルに2つ（全体再生 ＋ 行タップ）に集約する。専用のリピートボタンは設けない。
* **全体再生**: レッスン全体の会話を最初から最後まで通して自動再生できる。再生はブラウザの自動再生制限に従い、必ずユーザーの操作（再生ボタンのタップ等）を起点に開始する。一度開始した後は、各行を自動で繋いで連続再生する。
* **行タップで「その行から再生」**: 任意の行をタップ（クリック）すると、その行を再生開始位置として、以降の行を最後まで連続再生する。再生中の行はハイライト表示し、行の進行に合わせて自動でスクロールする。
* **タップ＝頭出し（リピートも兼ねる）**: 行タップは「その行の頭出し」として機能する。同じ行を続けてタップするたびに、その行の先頭から再生し直す。これにより**専用のリピートボタンなしで、聴き取れなかった行を何度でも繰り返せる**（シャドーイング用途を兼ねる）。
* **再生速度の変更**: 再生スピードを切り替えられる（例: `0.5x` / `0.75x` / `1.0x` / `1.25x`）。シャドーイングや聞き取り練習のため低速再生に対応する。
* 速度は音程を大きく崩さずに変更する（`playbackRate` の利用を想定）。
* 設定した速度はすべての再生（全体再生・行タップ再生）に共通で適用する。
* **再生中の速度切り替え**: 再生を止めずに速度を変更でき、変更後すぐ次の発話から（または即時）新しい速度が反映される。
* 選択した速度は `localStorage` に保存し、再訪時も維持する。
* 音声は `public/audio/` 配下の静的ファイルを直接参照して再生する。各行が独立した音声ファイルのため、任意の行を起点とした連続再生・繰り返しはクライアントだけで完結する（サーバー不要）。

## 5. システム処理フロー

```
【生成時（ローカルのみ）】
開発者 ➔ content/input/ に JSON を配置
                         │
                         ▼
             npm run generate（Node CLI スクリプト）
                         │
                         ├──➔ [kokoro-js (Node)] テキストを1行ずつ音声化し public/audio/ へ保存
                         └──➔ レッスンメタデータを content/lessons/ へ出力

【ビルド・デプロイ時】

                         ▼

【学習時（ブラウザ）】
ユーザー ➔ レッスン選択 ➔ レッスンページを表示（基本は事前生成）
                         │
                         ▼
             [画面表示] チャットUIでテキスト表示 ＆ public/audio/ の音声を再生
```

## 6. 非機能要件・制約事項

* **重い処理はビルド前に完結**: 音声生成はローカルのビルド前処理（CLI）で完結するため、本番のリクエスト処理は軽量で、ホスティングのサーバー実行時間制限（タイムアウト）の影響を受けない。サーバーサイド機能（SSR / API Routes）を使う場合も、重い処理を載せない方針とする。
* **データベース・外部ストレージ不要**: レッスンデータと音声をすべてリポジトリ内に保持するため、DB/Storage の無料枠容量や接続管理を考慮しなくてよい。
* **音声フォーマット（MP3）**: 配信・リポジトリ容量・ブラウザ互換性のバランスから、音声は **MP3** で保存する。人声主体のため、低めのビットレート（例: 48〜64kbps / mono / 24kHz 程度）でエンコードし、聞き取りに支障のない範囲で1ファイルを極力小さくする。
* **リポジトリ容量の節約**: 音声ファイルをリポジトリにコミットするため、上記の圧縮で容量を抑える。MP3 圧縮により基本的に Git LFS は不要となる想定だが、総容量が大きくなる場合は Git LFS の利用を検討する。
* **再現性**: 入力JSONと生成物を共にバージョン管理するため、任意の環境でビルド・デプロイが完全に再現可能。
* **Node.js バージョン**: `kokoro-js` および `fluent-ffmpeg` の動作要件から、Node.js **v18 以上**を必須とする。`.nvmrc` および `package.json` の `engines` フィールドで明示する。
* **ブラウザキャッシュ無効化**: 音声ファイルは同一パスのまま差し替わる可能性があるため、`<audio>` タグの `src` には `audio_hash` をクエリパラメータとして付与する（例: `/audio/at-the-cafe/1.mp3?v=<hash>`）。これによりハッシュが変わったターンのみブラウザが再取得する。
* **音声ファイル欠落時の挙動**: 再生時に参照先の `.mp3` が取得できない（HTTP エラー）ターンは、そのターンを**スキップ**して次のターンへ進む。スキップされた行は、プレイヤー上でエラー状態を示す控えめなインジケータ（例: 行番号の色変更）を表示する。
* **404 ページ**: 存在しないレッスンスラッグへのアクセス時は Next.js の `notFound()` を使用し、標準の 404 レスポンスを返す。
* **SEO**: 各レッスンページには `<title>` および `<meta name="description">` をレッスンの `title` / `description` から生成して設定する。OGP 画像は不要。

## 7. UI/デザイン要件

全体のルックは **Teenage Engineering**（OP-1 / PO シリーズ等）に通じる、**洗練されたミニマル＆機能主義（ファンクショナル）** なトーンで統一する。「装飾を削ぎ落とし、操作対象だけが際立つ」ハードウェア的な佇まいを目指す。

### 7.1. デザイン原則

* **ミニマル**: 余白を大胆にとり、要素を詰め込まない。1画面の情報量を絞り、操作対象（再生・行・トグル）を主役にする。
* **機能主義**: 装飾的なグラデーションや影、丸みの強い装飾は避け、要素の役割が形と配置で伝わるようにする。
* **グリッド**: 厳格なグリッド・整列に基づくレイアウト。余白とサイズに一貫したリズム（8px グリッド等）を持たせる。

### 7.2. カラーパレット

* **ベース**: ほぼモノクロ（オフホワイト／ライトグレー／ダークグレー〜ブラック）を基調とする。
* **アクセント**: アクセントカラーは **オレンジ `#FF5500`** に固定する。再生中の行・主要アクション・現在状態の強調にのみ限定使用し、これ以外の色は加えない（多色使いはしない）。
* ライト基調を標準とし、必要に応じてダークモードも同じ原則で用意できる構成にする。

### 7.3. タイポグラフィ

* **サンセリフ＋モノスペースの併用**: 本文・UIラベルはクリーンなサンセリフ。番号・話者名・ステータス・ボタンラベル等は**モノスペース**を用い、計器・機材的な印象を出す。
* ラベルは短く、必要に応じて**大文字＋トラッキング（字間広め）** を用いる。
* 英文（学習対象）は可読性最優先で十分な行間・文字サイズを確保する。

### 7.4. コンポーネント／操作系

* **ボタン**: 物理ボタンを思わせる角丸控えめ・フラットな矩形。アイコンは線の細い幾何学的なものに統一。再生／一時停止／リピート等は記号的アイコンで表現。
* **トグル**: 表示モード切替（英文＋訳／英文のみ）はハードウェアのスイッチを思わせる明快なトグルにする。
* **会話行**: チャットUIは過度な吹き出し装飾を避け、話者ごとに整列（1人目＝左／2人目＝右）・番号付け（`01` / `02`…）した端正なリスト／カードで表現。再生中の行はアクセントカラーで明確にハイライトする。
* **状態表示**: 再生位置・モード・進捗を、計器的でシンプルなインジケータ（小さなドット／バー／ラベル）で示す。

### 7.5. 実装方針

* **スタイルは `globals.css` で一括管理する**。カラー（アクセント `#FF5500` を含む）・フォント・余白スケール・角丸・状態色などのデザイントークンは、すべて `globals.css` 内の CSS カスタムプロパティ（`:root` の CSS 変数）として定義し、ここを唯一の変更点（Single Source of Truth）とする。
* Tailwind のテーマおよび shadcn/ui のコンポーネントは、`globals.css` で定義したトークン（CSS 変数）を参照するよう設定し、個別コンポーネントに色やサイズの値を直書きしない。デザイン調整は原則 `globals.css` の変更のみで完結させる。

### 7.6. レスポンシブ対応

PC・スマートフォンのどちらでも快適に学習できるよう、単一のUIでレスポンシブに対応する（PC専用／スマホ専用のページは作らない）。

* **モバイルファースト**: スマホでの学習を主要ユースケースとして設計し、画面幅に応じて段階的に拡張する（モバイル → タブレット → デスクトップ）。
* **流動的なレイアウト**: 会話のチャットUI・レッスン一覧（カード）は、画面幅に応じて1カラム⇄複数カラムや余白・文字サイズを調整し、PCの広い画面でも間延びせず、スマホでも詰まりすぎないようにする。Tailwind のブレークポイント（`sm` / `md` / `lg` 等）で制御する。
* **タッチとマウスの両対応**: スマホでは指で押しやすいタップ領域（最低 44px 程度）を確保し、PCではホバー状態やカーソル、キーボード操作にも対応する。
* **片手操作への配慮**: スマホでは主要操作（全体再生・速度切替・表示モード切替）を画面下部など親指の届きやすい位置に配置することを検討する。
* **可読性の確保**: 英文（学習対象）はどの画面サイズでも十分な文字サイズ・行間を保ち、長文や長い会話でも読みやすくする。
* **セーフエリア対応**: スマホのノッチ／ホームインジケータ等を避けるため、セーフエリア（`env(safe-area-inset-*)`）を考慮する。
* **動作確認**: 主要なビューポート（例: スマホ 375px幅 / タブレット 768px幅 / デスクトップ 1280px幅）で表示・操作が破綻しないことを確認する。

### 7.7. CSR によるインタラクション UX

CSR を活かし、ネイティブアプリのように軽快で途切れのない操作感を実現する。

* **即応性**: 再生／一時停止、行タップ再生（同じ行の再タップによる頭出し含む）、表示モード切替などの操作は、サーバー往復なしでクライアント状態のみで即座に反映する（体感遅延ゼロ）。
* **クライアントサイド遷移**: レッスン一覧⇄個別ページの移動は `next/link` によるソフトナビゲーションで行い、フルリロードせず SPA のように瞬時に切り替える。遷移時の不要なちらつき（フラッシュ）を避ける。
* **状態の永続化**: 表示モード（英文＋訳／英文のみ）・再生速度などのユーザー設定は `localStorage` に保存し、再訪時やページ遷移後も維持する。
* **再生状態の維持**: 表示モード切替やスクロール等の操作を行っても、再生位置・再生／一時停止状態・ハイライト中の行は途切れさせない（[再掲] 4.2 参照）。
* **滑らかな再生の繋ぎ**: 行ごとに独立した音声ファイルを連続再生する際は、次の行の音声を先読み（preload）して無音の隙間を最小化する。
* **ローディング表現**: 音声デコード待ちやモデル/データ読み込み中は、スピナーやスケルトンなどで状態を控えめに提示し、操作不能な「固まった」状態を作らない。
* **アクセシビリティ／入力**: 主要操作はキーボードでも行えるようにし、タップ領域はモバイルで十分な大きさを確保する。キーボードショートカットは以下のとおり。
  * `Space` — 再生 / 一時停止トグル
  * `→`（右矢印）— 次のターンへ進む（最終ターンでは停止）
  * `←`（左矢印）— 現在再生中のターンの頭出し（先頭ターンで押した場合もそのターンの頭出し）
* **翻訳切替時のレイアウト**: 日本語訳を非表示にする際はエリアを**完全に collapse**（高さゼロ）し、スペースを確保しない。これによりハイライト行の位置がシフトする場合があるが、それは許容する。
